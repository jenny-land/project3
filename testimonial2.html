01 Overview
Context
A self-initiated product design project. I designed a testimonial system to solve a gap in my own portfolio — there was no structured, trustworthy way to collect and present feedback. What began as a personal need quickly revealed a broader problem most platforms quietly ignore.
The Reframe
Most testimonial features are treated as static content — manually added, lightly curated, hard to maintain. This project reframes testimonials as a system, not a section: a structured, scalable way to collect, moderate, and present feedback while balancing trust, usability, and flexibility.
Approach
Rather than designing a single interface, I approached this as a full system design problem. I mapped the complete lifecycle of a testimonial — submission, validation, storage, moderation, display — ensuring that each part worked together rather than solving only the visible front-end layer.
Outcome
A complete, end-to-end testimonial system design. Full specification including data model, API structure, moderation workflow, design tokens, and UI components. A system that can be used within my portfolio and adapted for other designers, teams, or products.
02 The Problem

Most platforms treat testimonials as an afterthought. There's no consistent way to collect them, formatting and publishing are manual, validation and spam protection are minimal or absent, there's no moderation workflow, and there's no transparency between edited and original content.

The result is a system that is difficult to scale, hard to trust, and inconsistent for both users and administrators. Something that should build credibility ends up undermining it through its own structural gaps.

The core question this project set out to answer: how might we design a system that makes testimonials easy to collect, trustworthy to display, and sustainable to manage over time?

Solving that properly meant thinking beyond the UI layer — into content lifecycle, multiple actor types, status transitions, and the edge cases that don't show up until you look for them.

03 Approach

Rather than designing a single interface, I approached this as a full system design problem. That meant resisting the pull toward the visible layer — the form, the card, the carousel — until the underlying system logic was solid.

I mapped the complete lifecycle of a testimonial: submission, validation, storage, moderation, and display. Each stage has its own interface requirements, its own actor, and its own failure modes. Designing for the empty state — before any testimonials exist — is where most portfolio implementations quietly fail.

This ensured that each part of the experience worked together, rather than solving only the visible front-end layer. Form fields trace to database columns. Admin actions map to status transitions. Every decision connects back to the system, not just the screen.

The decision to build a system rather than add static content also reflected something broader: strong system thinking paired with modern tooling is a force multiplier. AI was used to accelerate execution — scaffolding the spec, stress-testing the data model. Every design decision and tradeoff was defined first. Direction came before delegation.

Design Tension · Trust vs. Friction
"Adding strict validation could improve trust — but at the cost of completion rates. The question is where the real safety net lives."
I chose lower-friction validation methods with lightweight anti-spam protections. The admin review layer is the actual safety net — invisible protection at zero UX cost to the submitter.
Design Principle · Dual Users
"The system supports two completely different people doing completely different things — and both experiences need to be right."
Designing for both the contributor and the admin shaped the form structure, the moderation tools, and the display rules. Neither user's experience was treated as secondary.
The Approach That Mattered

Treating content as a system rather than manually managing it means repeatable workflows, structured storage, and scalable logic — a system that grows without increasing maintenance complexity.

04 System Design
1
Submission Layer
Structured testimonial form with optional identity fields (name, role, organization), privacy controls (display name vs. full name), and clear consent and expectation setting. Designed to reduce friction while maintaining credibility.
2
Validation & Trust
Required fields for meaningful input. Optional credibility indicators. Anti-spam strategy using a low-friction approach. The key decision: balancing ease of submission with trustworthiness of content.
3
Data Structure
Defined schema supporting flexible testimonial types. Strict separation between raw submission data and edited display content — allowing refinement without losing the original voice.
4
Moderation Workflow
Admin dashboard for review and approval. Ability to approve, edit, reject, and feature testimonials. This transforms testimonials from static content into a fully manageable system.
Content lifecycle — Submission → Validation → Storage → Moderation → Display

The full content lifecycle mapped before any UI work began. Five stages, two actors, one system.

The Challenge
Dynamic rendering based on testimonial status — approved vs. featured — requires clear hierarchy and consistent formatting. How do you give the admin meaningful editorial control without building a CMS from scratch?
The Decision
Display logic is driven entirely by the data model. An is_featured boolean controls elevated placement. A status enum governs visibility. The admin doesn't need to touch layout — they approve, feature, or reject, and the system handles presentation.
When the data model is right, the display layer almost designs itself. Structure is the design.
05 Key Design Decisions

The most consequential decisions in this project weren't about visual design — they were about trust architecture, data integrity, and the ethics of editing someone else's words. Each decision below was made deliberately, with explicit rationale.

Designing for Trust, Not Just Collection
"Many systems prioritize collecting testimonials quickly. I focused on ensuring they would feel credible and intentional when displayed."
This included separating raw vs. edited content, allowing optional identity verification signals, and maintaining transparency in how testimonials are presented. Trust is not a visual layer — it's built through structure and process.
Reducing Friction Without Losing Quality
"Email verification could improve trust, but breaks momentum after emotional intent is established — at exactly the wrong moment."
I chose lower-friction validation with lightweight anti-spam protections. This keeps the system usable while still maintaining reasonable trust signals. The admin review layer is the real safety net.
The Challenge
Editorial control is necessary — but editing someone else's words raises an ethical question. How do you refine a testimonial for display without erasing the person's original voice? And how do you make that distinction visible and auditable?
The Decision
Two content fields in the database schema: content_raw stores the original submission, untouched. content_edited stores the polished version for display. Keeping both means editorial control without data loss — the submitter's original voice is always preserved, even when the displayed version is refined.
Transparency in how testimonials are handled is itself a trust signal. The system should be auditable by design.

06 AI-Assisted Workflow

AI was used as a productivity and structuring tool — not a decision-maker. I used it to help structure database schema and system logic, generate and refine API endpoint patterns, and validate the completeness of flows and edge cases.

All outputs were reviewed, adjusted, and aligned with the intended user experience and system behavior. This allowed me to move more efficiently through technical structuring while maintaining full control over design decisions. Direction came before delegation.

On AI and Design Authorship

The value of AI in this workflow was in acceleration, not authorship. Every system decision, UX tradeoff, and design rationale was defined first — then AI was used to execute against that foundation more efficiently. The thinking was always mine.
