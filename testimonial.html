02 The Problem

Portfolios live or die on trust. Hiring managers want to know what it's actually like to work with you — not just what you made. Testimonials are one of the most direct answers, and mine were absent.

I could have typed two quotes into static HTML and called it done. But that immediately raised questions I couldn't ignore: How do new testimonials get added? How do I maintain quality without losing the submitter's original voice? What does the page look like before I have any? How do I protect people who want to stay anonymous?

Those questions aren't content questions. They're product questions — about a content lifecycle, multiple actors, and real edge cases. Solving the empty space properly meant designing the system before touching the UI.

The pipeline has six stages: Collect → Validate → Store → Review → Approve → Display. Each stage has its own interface requirements, its own actor (submitter or admin), and its own edge cases. Designing for the empty state — before any testimonials exist — is where most portfolio implementations quietly fail.

The decision to build a system rather than add static content also reflected something broader: strong system thinking paired with modern tooling is a force multiplier. AI was used to accelerate execution — scaffolding the spec, stress-testing the data model, building the prototype. Every design decision and tradeoff was defined first. Direction came before delegation.

The Design Reframe

The empty space on my portfolio wasn't a design problem. It was a product problem — a missing content lifecycle with no system to sustain it. Solving it properly meant thinking through the full pipeline before writing a single line of UI code.

03 System Thinking

Before designing any interface, I mapped the end-to-end pipeline. A testimonial system is a lightweight CMS — it has content states, multiple actors, status transitions, and edge cases that don't show up until you look for them.

Defining the pipeline first meant every downstream decision had clear constraints. Form fields map directly to database columns. Admin actions map to status transitions. The empty state is a designed state, not an oversight. The data model was designed alongside the UI, not after it — which is why an engineer could pick up the spec and build directly from it.

Two fields in the database schema capture the core ethical tension of the project: content_raw stores the original submission, untouched. content_edited stores the polished version for display. Keeping both means editorial control without data loss — the submitter's original voice is always preserved, even when the displayed version is refined.

The is_featured boolean and display_name field trace back directly to UX decisions: featured testimonials get elevated placement, and submitters choose their own display name — full name, first name only, or anonymous. Each field earns its place by mapping to a specific user need or admin action.

Design Tension · Trust vs. Friction
"Email verification increases legitimacy — but breaks momentum at exactly the wrong moment, after emotional intent is established but before the task is complete."
V1 decision: honeypot field + manual moderation instead of email verification. The admin review layer is the real safety net. Invisible protection, zero UX cost to the submitter. V2 can add verification once real spam justifies the friction.
Design Principle · Display Component
"A card signals: this is a widget, apart from the page. For testimonials that's exactly wrong. Typography and whitespace should carry the design."
The display component has no card container — intentionally. Editorial integration over component isolation. The words carry the weight. This decision was made before a single line of display CSS was written.
Key Design Insight

When a designer hands over a clear data model alongside wireframes, the engineering conversation changes entirely. Designing the system before the screens meant the spec was complete — not just the visuals.

The Challenge
Prompt chips — small clickable phrases that append starter text to the testimonial field — help submitters past blank-page anxiety. But placeholder text in the textarea disappears the moment someone starts typing, removing the scaffold at exactly the moment it's needed. How do you guide without prescribing?
The Decision
Chips stay visible after interaction — they're not placeholder text, they're an active writing tool embedded in the form. Clicking a chip appends a starter phrase to whatever the submitter has already written. The chip toggles active/inactive so the submitter can see what they've used. The textarea remains fully editable — chips prompt without prescribing.
The distinction between a scaffold and a constraint is whether the user retains their own voice. Chips scaffold. They don't constrain.
05 Final Design

The submission form is designed to reduce hesitation and increase quality simultaneously. Every field was chosen deliberately — to lower the barrier to starting, protect submitters who want privacy, and give the admin enough context to make good moderation decisions. The form ends with a "Cheers!" success state — champagne glasses, a warm thank-you, and clear expectations about next steps. Submission should feel like a moment worth marking.

The admin dashboard gives me filter, select, edit, approve, feature, and reject controls in one interface. Raw and edited content are shown side by side — editorial control without losing the original voice. The display component on the portfolio itself has no card border by design: testimonials are integrated into the page typographically, not isolated as a widget.

What this taught me

The most important design work often happens before the UI exists — in the questions you ask, the constraints you name, the tradeoffs you make explicit before a single pixel is placed. This project was a deliberate practice of that principle.

What Worked
Designing the data model before the UI forced every interface decision to be grounded. When I asked "what fields does the form need?" the answer came from the schema, not from intuition. The form and the database ended up in complete alignment — which is rarer than it should be.
What I'd Change
The one gap in the current spec is a notification strategy — how does the admin know when a new submission arrives? A lightweight email alert would close the loop without adding system complexity. That's V1.1, and it should have been in V1. I'll add it before implementation begins.
Next Steps
Implement the system — starting with the submission form and admin dashboard using the existing spec. Run a moderated usability test on the submission form with 5 potential submitters before going live. Add the admin notification layer. Then: collect real testimonials and let the system prove itself.
